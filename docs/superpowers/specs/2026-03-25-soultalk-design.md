# 心语（SoulTalk）— 设计文档

> AI 情感陪伴助手，像给朋友打电话一样自然的语音聊天体验

## 背景

用户希望开发一个手机网页版的情感陪伴助手，使用 MiniMax-M2.7 模型，支持中英文实时语音对话。核心体验是"像给朋友打电话"，AI 根据用户情绪自适应沟通风格，越聊越懂你。

## 产品定义

**产品名**：心语（SoulTalk）

**一句话定位**：一个能用语音聊天的 AI 情感陪伴助手，像给朋友打电话一样自然。

**核心角色**：**Harry** — 你的 AI 朋友
- 全能型朋友，没有固定风格标签
- 根据用户当前情绪自动切换：难过时温暖共情、纠结时理性分析、无聊时幽默陪聊
- 随着对话积累，逐渐记住用户的性格偏好、表达习惯、敏感话题
- 越聊越像"懂你的老朋友"

**目标用户**：
- 独居年轻人（18-35 岁），需要日常陪伴和倾诉出口
- 情绪低落时需要即时安慰的人
- 社恐/不善表达的人，语音比打字更舒适

**语言支持**：中文 + 英文（自动识别，不用手动切换）

**MVP 范围**：
- 单角色（Harry）自适应风格
- 实时语音对话（STT → LLM → TTS）
- 会话内上下文记忆（最近 20 轮）
- 跨会话用户画像（localStorage）
- 中英文自动识别
- 移动端优先设计

**MVP 不做**：
- 不做用户注册/登录（匿名使用）
- 不做云端数据同步
- 不做聊天历史跨会话保存
- 不做自定义角色
- 不做付费功能

## 技术方案

**架构**：纯前端 SPA，无需后端服务器

```
用户说话 → Web Audio 录音
         → Web Speech API（语音转文字，浏览器原生）
         → MiniMax M2.7 LLM（生成回复）
         → MiniMax TTS API（文字转语音）
         → 播放语音给用户
```

**前端**：
- React + TailwindCSS
- 移动优先设计，竖屏全屏体验
- Vite 构建，Netlify 部署

**语音录入（STT）**：
- 浏览器原生 Web Speech API（免费、延迟低、支持中英文）
- 自动检测语言，不需用户手动切换

**AI 对话（LLM）**：
- MiniMax M2.7（Anthropic 兼容接口）
- System Prompt 包含：角色设定 + 用户画像 + 风格自适应规则
- 流式输出（边生成边送给 TTS，减少等待）

**语音合成（TTS）**：
- MiniMax T2A v2 API（支持中英文混合，情感自然）
- 温暖自然的音色
- 流式播放：收到一段就播一段

**部署**：
- Vercel 或 Netlify（免费）
- Vite 开发代理绕过 CORS（同 AcceptBot 方案）

**延迟预估**：
- Web Speech API 识别：~0.5 秒
- LLM 首 token：~1 秒
- TTS 首段音频：~0.5 秒
- 总延迟约 2 秒（流式优化后）

**月度成本**：
- MiniMax M2.7 LLM：按量付费
- MiniMax TTS：按字符计费
- 部署：免费
- 预估：轻度使用 ¥20-50/月

## 交互设计

### 页面结构（2 个页面）

**1. 欢迎页（首次进入）**：
- 全屏深色背景，中心显示 Harry 头像/图标
- 「嗨，我是 Harry，你的 AI 朋友」
- 「点击开始聊天」按钮
- 底部小字：隐私说明链接

**2. 对话页（核心页面）**：
- 顶部：Harry 名字 + 状态指示（聆听中/思考中/说话中）
- 中间：对话气泡区
  - 用户消息靠右（语音波形 + 转写文字）
  - Harry 回复靠左（头像 + 文字 + 播放按钮）
  - 背景色随对话氛围微妙变化（暖色=积极，冷色=低落，中性=日常）
- 底部：语音交互区
  - 大圆按钮（按住说话）
  - 按住时显示声波动画
  - 松开 → 识别 → 思考 → 语音播放
  - 旁边有文字输入切换按钮

### 交互流程

```
空闲状态（大圆按钮）
  ↓ 按住
录音中（声波动画，按钮变色）
  ↓ 松开
识别中（「听到了...」）
  ↓ 0.5s
思考中（Harry 头像微动画）
  ↓ 1-2s
播放语音（Harry 说话，文字同步显示）
  ↓ 播完
空闲状态
```

### 手势交互
- 按住说话，松开发送（微信习惯）
- 上滑取消发送
- 点击历史语音可重新播放

### 移动端适配
- 竖屏全屏设计
- 底部按钮区域拇指友好
- 键盘弹出时对话区上推
- 支持 PWA 添加到主屏幕

## 记忆系统

### 用户画像结构（localStorage）

```json
{
  "name": "未知",
  "basicInfo": {
    "age": null,
    "job": null,
    "city": null,
    "interests": []
  },
  "personality": {
    "preferredStyle": "balanced",
    "communicationMode": "casual",
    "humorLevel": "medium"
  },
  "emotionalPatterns": {
    "commonTopics": [],
    "triggers": [],
    "comfortMethods": []
  },
  "importantMemories": [],
  "sessionCount": 0,
  "lastSessionDate": null
}
```

### 画像更新机制
- 触发时机：对话结束（关闭页面或 5 分钟无操作）
- 方式：让 LLM 从本次对话提取新信息，和已有画像合并
- Prompt：「根据这段对话，更新用户画像 JSON，只更新有新信息的字段」

### 画像如何影响对话
- 注入 System Prompt：「用户叫小明，在北京做程序员，喜欢跑步。上次聊到他和女朋友吵架了。他偏好直接的沟通方式。」
- Harry 开场白自适应：
  - 首次：「嗨！我是 Harry，很高兴认识你」
  - 回访：「小明，好久不见！上次说的事情后来怎么样了？」
  - 检测到低落：「感觉你今天心情不太好，想聊聊吗？」

### 隐私保护
- 所有数据仅存浏览器本地
- 提供「清除记忆」按钮
- 不收集任何可识别个人身份的信息
- 明确告知用户数据存储方式

## 时间线与里程碑

| 阶段 | 时间 | 目标 |
|------|------|------|
| 准备 | 第 0 周 | 确认 MiniMax TTS API 可用，测试语音合成效果 |
| 核心对话 | 第 1-2 周 | 文字对话 MVP：Harry 角色 + 自适应 Prompt + 记忆系统 |
| 语音输入 | 第 3 周 | Web Speech API 录音 → 文字 → LLM 回复 |
| 语音输出 | 第 4-5 周 | MiniMax TTS 集成，流式播放，状态动画 |
| UI 打磨 | 第 6 周 | 移动端适配、声波动画、情绪背景色、交互细节 |
| 记忆系统 | 第 7 周 | 跨会话画像提取与注入，回访体验 |
| 测试上线 | 第 8 周 | 全流程测试，部署，分享给朋友试用 |

**每周 5 小时分配**：
- 开发周：5 小时全部用于搭建
- 测试周：3 小时开发 + 2 小时真机测试

**成功标准**：
- 8 周内上线可用产品
- 语音对话延迟 < 3 秒
- 5 个朋友试用，至少 3 个愿意再用第二次
- 8 周后无人使用 → 复盘调整
