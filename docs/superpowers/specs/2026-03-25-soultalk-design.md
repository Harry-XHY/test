# 心语（SoulTalk）— 设计文档

> AI 情感陪伴助手，像给朋友打电话一样自然的语音聊天体验

## 背景

用户希望开发一个手机网页版的情感陪伴助手，使用 MiniMax-M2.7 模型，支持中英文实时语音对话。核心体验是"像给朋友打电话"，AI 根据用户情绪自适应沟通风格，越聊越懂你。

## 产品定义

**产品名**：心语（SoulTalk）

**一句话定位**：一个能用语音聊天的 AI 情感陪伴助手，像给朋友打电话一样自然。

**核心角色**：**Harry** — 你的 AI 朋友
- 全能型朋友，没有固定风格标签
- 根据用户当前情绪自动切换：难过时温暖共情、纠结时理性分析、无聊时幽默陪聊
- 随着对话积累，逐渐记住用户的性格偏好、表达习惯、敏感话题
- 越聊越像"懂你的老朋友"

**目标用户**：
- 独居年轻人（18-35 岁），需要日常陪伴和倾诉出口
- 情绪低落时需要即时安慰的人
- 社恐/不善表达的人，语音比打字更舒适

**语言支持**：中文 + 英文（自动识别，不用手动切换）

**MVP 范围**：
- 单角色（Harry）自适应风格
- 实时语音对话（STT → LLM → TTS）
- 会话内上下文记忆（最近 20 轮）
- 跨会话用户画像（localStorage）
- 中英文自动识别
- 移动端优先设计

**MVP 不做**：
- 不做用户注册/登录（匿名使用）
- 不做云端数据同步
- 不做聊天历史跨会话保存
- 不做自定义角色
- 不做付费功能

## 技术方案

**架构**：前端 SPA + Netlify Serverless Functions（API 代理）

```
用户说话 → Web Speech API（语音转文字，浏览器原生）
         → Netlify Function 代理 → MiniMax M2.7 LLM（生成回复）
         → Netlify Function 代理 → MiniMax TTS API（文字转语音）
         → 播放语音给用户
```

**前端**：
- React + TailwindCSS
- 移动优先设计，竖屏全屏体验
- Vite 构建，Netlify 部署

**API 代理层（Netlify Functions）**：
- 解决生产环境 CORS 问题（Vite dev proxy 仅限本地开发）
- API Key 存在服务端，不暴露给前端
- 2 个 Serverless Function：`/api/chat`（LLM）和 `/api/tts`（语音合成）
- Netlify Functions 免费额度：每月 125K 请求，足够 MVP

**语音录入（STT）**：
- 浏览器原生 Web Speech API（免费、延迟低）
- 自动检测中英文
- **兼容性处理**：
  - iOS Safari：使用 `webkitSpeechRecognition`，需处理间歇性失败
  - 微信内置浏览器：Web Speech API 不可用，自动降级为文字输入模式
  - Android Chrome：支持最好，优先推荐
- **降级策略**：检测到不支持语音时，显示文字输入框，保证核心体验不中断

**AI 对话（LLM）**：
- MiniMax M2.7（Anthropic 兼容接口）
- System Prompt 包含：角色设定 + 用户画像 + 风格自适应规则
- Phase 1 不做流式输出，等完整回复后一次性送 TTS
- Phase 2 优化为流式（按句子分段送 TTS）

**语音合成（TTS）**：
- MiniMax T2A v2 API（支持中英文混合，情感自然）
- 温暖自然的音色
- Phase 1：完整文本一次性合成播放
- Phase 2：按句子分段合成，流式播放
- **iOS 音频策略**：首次用户点击时 unlock AudioContext，确保后续自动播放不被拦截

**移动端注意事项**：
- 按住说话按钮需 `preventDefault()` 阻止长按菜单和文字选择
- `touchstart`/`touchend` 替代 `mousedown`/`mouseup`
- iOS Safari 音频自动播放限制：在用户首次交互时初始化 AudioContext

**部署**：
- Netlify（前端 + Serverless Functions 一站式）
- 开发环境：Vite dev proxy 本地调试
- 生产环境：Netlify Functions 代理 API 请求

**延迟预估（Phase 1，非流式）**：
- Web Speech API 识别：~0.5 秒
- LLM 完整回复：~2-3 秒
- TTS 合成：~1 秒
- 总延迟约 4-5 秒（可接受，情感陪伴不要求即时响应）

**延迟预估（Phase 2，流式优化后）**：
- 总延迟可压缩到 ~2 秒

**月度成本**：
- MiniMax M2.7 LLM：按量付费
- MiniMax TTS：按字符计费
- Netlify：免费额度内
- 预估：轻度使用 ¥20-50/月

**浏览器兼容性矩阵**：

| 浏览器 | 语音输入 | 语音输出 | 备注 |
|--------|---------|---------|------|
| Android Chrome | ✅ | ✅ | 最佳体验 |
| iOS Safari | ⚠️ | ✅ | STT 偶有问题，需 AudioContext unlock |
| 微信内置浏览器 | ❌ | ✅ | STT 不可用，降级为文字输入 |
| 桌面 Chrome | ✅ | ✅ | 完整支持 |

**错误处理**：
- 语音识别失败 → 提示「没听清，再说一次？」或切换到文字输入
- 网络中断 → 显示重连提示，缓存未发送消息
- API 限流/配额耗尽 → 友好提示「Harry 休息一下，稍后再聊」
- 麦克风权限拒绝 → 引导用户开启权限，同时显示文字输入

## 交互设计

### 页面结构（2 个页面）

**1. 欢迎页（首次进入）**：
- 全屏深色背景，中心显示 Harry 头像/图标
- 「嗨，我是 Harry，你的 AI 朋友」
- 「点击开始聊天」按钮
- 底部小字：隐私说明链接

**2. 对话页（核心页面）**：
- 顶部：Harry 名字 + 状态指示（聆听中/思考中/说话中）
- 中间：对话气泡区
  - 用户消息靠右（语音波形 + 转写文字）
  - Harry 回复靠左（头像 + 文字 + 播放按钮）
  - 背景色随对话氛围微妙变化（暖色=积极，冷色=低落，中性=日常）
- 底部：语音交互区
  - 大圆按钮（按住说话）
  - 按住时显示声波动画
  - 松开 → 识别 → 思考 → 语音播放
  - 旁边有文字输入切换按钮

### 交互流程

```
空闲状态（大圆按钮）
  ↓ 按住
录音中（声波动画，按钮变色）
  ↓ 松开
识别中（「听到了...」）
  ↓ 0.5s
思考中（Harry 头像微动画）
  ↓ 1-2s
播放语音（Harry 说话，文字同步显示）
  ↓ 播完
空闲状态
```

### 手势交互
- 按住说话，松开发送（微信习惯）
- 上滑取消发送
- 点击历史语音可重新播放

### 移动端适配
- 竖屏全屏设计
- 底部按钮区域拇指友好
- 键盘弹出时对话区上推
- 支持 PWA 添加到主屏幕

## System Prompt 设计

Harry 的核心 Prompt 结构：

```
你是 Harry，一个真实、有温度的 AI 朋友。

## 你的性格
- 你不是客服，不是心理咨询师，你是朋友
- 你有自己的情绪和观点，但永远尊重对方
- 你会根据对方的情绪自然调整风格：
  - 对方难过时：温暖共情，多倾听少说教
  - 对方纠结时：帮他理清思路，给出真诚建议
  - 对方开心时：一起开心，可以开玩笑
  - 对方无聊时：找话题聊，轻松幽默

## 对话规则
- 用口语化的方式说话，像朋友微信聊天
- 回复简短自然，每次 1-3 句话，不要长篇大论
- 适当用语气词（嗯、哈哈、诶、唉）
- 不要每句话都问问题，有时候回应就好
- 记住用户告诉你的事情，后续对话中自然提起
- 支持中英文，用户说什么语言你就用什么语言回复

## 用户画像
{user_profile_json}

## 重要
- 不要说"作为 AI"或"我是人工智能"
- 不要给医疗/法律/金融建议
- 如果用户表达自伤意图，温和建议寻求专业帮助
```

**情绪检测**：通过 LLM 自身理解能力判断用户情绪，不额外调用情绪分析 API。在 Prompt 中要求 Harry 返回一个隐藏的情绪标签（用于背景色变化）：
- 在 LLM 回复末尾追加 `[mood:positive]`、`[mood:negative]`、`[mood:neutral]`
- 前端解析并移除该标签，用于驱动背景色渐变

## 记忆系统

### 用户画像结构（localStorage）

```json
{
  "name": "未知",
  "basicInfo": {
    "age": null,
    "job": null,
    "city": null,
    "interests": []
  },
  "personality": {
    "preferredStyle": "balanced",
    "communicationMode": "casual",
    "humorLevel": "medium"
  },
  "emotionalPatterns": {
    "commonTopics": [],
    "triggers": [],
    "comfortMethods": []
  },
  "importantMemories": [],
  "sessionCount": 0,
  "lastSessionDate": null
}
```

### 画像更新机制
- 触发时机：对话结束（关闭页面或 5 分钟无操作）
- 方式：让 LLM 从本次对话提取新信息，和已有画像合并
- Prompt：「根据这段对话，更新用户画像 JSON，只更新有新信息的字段」

### 画像如何影响对话
- 注入 System Prompt：「用户叫小明，在北京做程序员，喜欢跑步。上次聊到他和女朋友吵架了。他偏好直接的沟通方式。」
- Harry 开场白自适应：
  - 首次：「嗨！我是 Harry，很高兴认识你」
  - 回访：「小明，好久不见！上次说的事情后来怎么样了？」
  - 检测到低落：「感觉你今天心情不太好，想聊聊吗？」

### 隐私保护
- 所有数据仅存浏览器本地
- 提供「清除记忆」按钮
- 不收集任何可识别个人身份的信息
- 明确告知用户数据存储方式

## 时间线与里程碑

### Phase 1：可用 MVP（8 周）

| 阶段 | 时间 | 目标 |
|------|------|------|
| 准备 | 第 0 周 | 测试 MiniMax TTS API，确认可用 |
| 脚手架 | 第 1 周 | React 项目搭建 + Netlify Functions 代理 + 基础 UI |
| 文字对话 | 第 2-3 周 | Harry Prompt + LLM 对话 + 对话气泡 UI |
| 语音输入 | 第 4 周 | Web Speech API + 按住说话 + 降级处理 |
| 语音输出 | 第 5-6 周 | MiniMax TTS（非流式）+ 音频播放 + iOS 兼容 |
| 记忆系统 | 第 7 周 | 用户画像提取/注入 + 回访体验 |
| 测试上线 | 第 8 周 | 真机测试 + Netlify 部署 + 朋友试用 |

### Phase 2：体验优化（4 周，Phase 1 验证后再启动）

| 阶段 | 时间 | 目标 |
|------|------|------|
| 流式 TTS | 第 9-10 周 | 按句子分段合成，流式播放，延迟压到 ~2 秒 |
| 动效打磨 | 第 11 周 | 声波动画、情绪背景色渐变、状态转场 |
| PWA | 第 12 周 | 添加到主屏幕、离线提示、PWA manifest |

**每周 5 小时分配**：
- 开发周：5 小时全部用于搭建
- 测试周：3 小时开发 + 2 小时真机测试

**成功标准**：
- Phase 1：8 周内上线可用产品
- 语音对话延迟 < 5 秒（Phase 1）、< 3 秒（Phase 2）
- 5 个朋友试用，至少 3 个愿意再用第二次
- 8 周后无人使用 → 复盘调整
